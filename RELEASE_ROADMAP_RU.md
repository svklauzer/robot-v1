# Дорожная карта ускоренного релиза: Trading Robot + Telegram Contour

Дата анализа: 25 мая 2026 (UTC)

## 1) Что уже есть в проекте (техническая готовность)

### Архитектура
- **Инфра через Docker Compose**: `db (Postgres 16)`, `redis (7)`, `api (FastAPI)`, `web (Next.js)`. 
- Контуры уже поднимаются вместе и экспонируют порты 5432/6379/8000/3000.

### Backend (API + бизнес-логика)
- Инициализация системы и фоновых циклов в `apps/api/main.py`:
  - `background_robot_loop` (основной торговый цикл, 1 мин интервал).
  - `background_subscription_loop` (контроль подписок, 1 раз в 6 часов).
- Основной цикл в `apps/api/workers/robot_loop.py`:
  - сбор/оценка сигнала (`market_intelligence`),
  - фильтрация качества,
  - построение трейд-плана,
  - risk/exposure проверки,
  - публикация в Telegram,
  - lifecycle сопровождение позиций.

### Telegram контур
- Реализован отдельный маршрутизатор `apps/api/services/telegram_router.py`:
  - Обязательная отправка VIP full signal (ошибка — критична).
  - Опциональная отправка FREE teaser/update.
  - Разделение по grade (A+/A/B/C) и разным каналам.
- Есть сервис `SubscriptionWatchdog` для статусов подписок (active/trial/expired).

### Web контур
- Есть клиент API (`apps/web/lib/api.ts`) и dashboard-компоненты (`HealthPanel`, `SignalsTable`, `PositionsTable`, `EquityChart` и т.д.), что позволяет быстро довести продукт до «операционного кабинета».

## 3) Карта взаимодействия сервисов (as-is)

1. `RobotLoop.step()` анализирует символы, фильтры и качество.
2. При одобрении создает сигнал + рассчитывает план и risk limits.
3. `SignalLifecycleManager` сопровождает сигнал/позицию по жизненному циклу.
4. `TelegramRouter` публикует:
   - VIP: полный сигнал + сопровождение,
   - FREE: teaser/часть апдейтов.
5. `SubscriptionWatchdog` валидирует жизненный цикл подписчиков (просрочки/напоминания).
6. `web` читает API и отображает статус/сигналы/позиции/здоровье.

---

## 4) Главные GAP’ы до релиза с продажами

1. **Монетизация не замкнута end-to-end**:
   - нет подтвержденного платежного процесса,
   - нет автопродления/реактивации,
   - нет автоматического управления доступом в VIP по факту оплаты.
2. **Bot menu и UX Telegram** не доведены до funnel-режима:
   - стартовое меню, тарифы, FAQ, поддержка, статус подписки, продление.
3. **Торговая стабильность** пока слабая для агрессивного масштабирования трафика.
4. **Операционный мониторинг и SLA** требуют упаковки (алерты по отказам отправки, latency, просадки и т.п.).

## 5) Дорожная карта быстрого запуска (6 недель)

## Фаза 0 — 72 часа: Release freeze и «красные кнопки»
**Цель:** ничего не ломаем, фиксируем релизный baseline.
- Заморозить scope: только критичные задачи для GTM.
- Ввести release-ветку и критерии go/no-go.
- Проверить rollback-процедуру по API/web/docker.
- Добавить ежедневный owner-report: PnL, winrate, signal failure reasons, telegram send failures.

**Критерий выхода:** повторяемый деплой + наблюдаемость + аварийный откат.

## Фаза 1 — Неделя 1: Монетизация ядра
**Цель:** начать принимать оплату и автоматически выдавать доступ.
- Реализовать Payment Adapter (провайдер X) + webhook обработчик.
- Состояния платежа: `created -> paid -> failed -> refunded`.
- Таблицы/модели: платежи, транзакции, idempotency key.
- После `paid`:
  - создание/продление `Subscriber`,
  - действие в Telegram: выдача/подтверждение доступа VIP,
  - уведомления пользователю + владельцу.
- Антидубль вебхуков и журнал аудита.

**Критерий выхода:** E2E тест “платеж -> доступ VIP -> подтверждение в боте”.

## Фаза 2 — Неделя 2: Telegram меню и воронка
**Цель:** собрать управляемый продуктовый интерфейс в боте.
- Меню: `Старт`, `Тарифы`, `Оплатить`, `Проверить подписку`, `Продлить`, `FAQ`, `Поддержка`, `Условия`.
- Гейт premium-команд по статусу подписки.
- Сценарии trial (если нужен): автоистечение и upsell в VIP.
- Шаблоны сообщений: onboarding, payment success/fail, renewal reminders.

**Критерий выхода:** пользователь без оператора проходит путь onboarding->оплата->получение доступа.

## Фаза 3 — Неделя 3: Стабилизация торговой модели перед масштабом
**Цель:** снизить репутационные риски при запуске трафика.
- Тактические изменения по аналитике:
  - ужесточить правила `failed_setup_exit` (тайм-аут сетапа, более ранний invalidate);
  - усилить фиксацию частичной прибыли и/или трал после положительного хода;
  - ограничить худшие символы по net pnl (динамический denylist/пониженный вес).
- A/B на небольшом объеме сигналов (paper/live-sim) с daily review.

**Критерий выхода:** улучшение net pnl trend и снижение доли `failed_setup_exit`.

## Фаза 4 — Неделя 4: Операционная готовность и поддержка
**Цель:** обеспечить предсказуемость сервиса после релиза.
- SLO/SLA: доступность API, latency Telegram send, error budget.
- Алерты: падение winrate, всплеск send_failed, long downtime loop.
- Runbook: «что делать при…» (платежи, телеграм ошибки, деградация стратегии).
- Мини-CRM поток: обращения, возвраты, ручное продление, blacklist/fraud flags.

**Критерий выхода:** on-call может вести систему по инструкциям без разработчика.

## Фаза 5 — Неделя 5-6: Soft launch и масштабирование
**Цель:** запуститься быстро, но контролируемо.
- Soft launch на ограниченную аудиторию.
- KPI: конверсия оплаты, churn, refund rate, retention D7/D30, ARPU, support load.
- Корректировки оффера/прайсинга/контента FREE→VIP по факту данных.
- Затем постепенное увеличение трафика.

**Критерий выхода:** unit-экономика и операционная нагрузка под контролем.

## 7) Конкретные KPI на релиз (первый месяц)

- Технические:
  - uptime API/Web > 99%.
  - успешность Telegram VIP send > 99.5%.
  - webhook processing success > 99.9% (с учетом ретраев).
- Торговые:
  - снижение доли `failed_setup_exit` минимум на 25% от текущего baseline.
  - рост доли сделок с зафиксированной прибылью.
- Продукт/монетизация:
  - конверсия FREE→оплата,
  - renew rate,
  - refund rate,
  - payback периода acquisition.

## 8) Риски и как их закрыть

- **Риск №1: нестабильный PnL при активном маркетинге**
  - Митигировать soft launch + risk throttle + daily quality gates.
- **Риск №2: сбои оплаты/доступа в VIP**
  - Митигировать idempotency, retries, reconciliation-job каждые N минут.
- **Риск №3: перегрузка support**
  - Митигировать self-service меню, FAQ, статус подписки, автоуведомления.
