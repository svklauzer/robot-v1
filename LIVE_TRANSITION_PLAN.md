# План перехода в live

Дата: 2026-07-07. Цель — довести систему от paper до live_trade без слива депозита,
проходя через воротные проверки (gates). На каждой фазе есть критерий перехода и
kill-switch. Не перепрыгивать фазы.

Контекст: сейчас `TRADING_MODE=paper_signal`, `ENABLE_LIVE_ORDERS=False`,
`ML_MODE=advisory` (рантайм), winrate 28.9% на 197 сделках. Перед live должны
отработать фиксы: ML-гейт (skip блокирует вход), ранний trend_peak_lock, фильтр
потока на входе, фикс ML-множителя на 1.0.

---

## Фаза 0 — Санити (0.5 дня)

Прежде чем что-либо включать:

- `cd apps/api && python -m pytest -q` — все тесты зелёные (я правил test_exit_policy).
- `python -c "import core.config, services.exit_policy, services.ml_controller, workers.robot_loop"` — импорт без ошибок.
- Дёрнуть `GET /intelligence/scan`, `/orderbook/state`, `/ml/outcomes/stats` — движки живы, данные свежие.

**Gate 0:** тесты зелёные, импорт чистый, эндпоинты отвечают. Иначе — стоп.

---

## Фаза 1 — Paper с новой логикой (5–7 дней, ≥40 закрытых сделок)

Собираем статистику на исправленной системе. Ничего живого.

Env:
```
TRADING_MODE=paper_signal
ENABLE_LIVE_ORDERS=false
ML_MODE=advisory                 # skip теперь реально блокирует (ML_ENFORCE_ADVISORY_SKIP=true)
ML_SIZE_SCALING_ENABLED=false    # множитель зафиксирован на 1.0
ENABLE_ORDERBOOK_ENGINE=true
ENABLE_FUTURES_EXECUTION=true    # нужно для шортов (execution_market_type=swap)
ALLOW_SHORTS=true
```

Что смотреть в телеметрии ежедневно:
- доля `positive_then_negative` среди закрытых — должна упасть c ~95% заметно (цель <60%);
- новые коды `blocked_by_ml_advisory_skip`, `trend_peak_lock`, `depth_cvd_*` — реально срабатывают;
- winrate и суммарный net PnL на новой выборке;
- баланс long/short (шорты должны появиться, раз рынок катится вниз).

**Gate 1 (все одновременно):**
- ≥40 закрытых сделок на новой логике;
- net PnL за окно ≥ 0 (не слив);
- `positive_then_negative` < 60%;
- нет незакрытых «зависших» позиций и крашей manage-loop.

Kill-switch: если net PnL за 3 дня < −5% эквити — стоп, разбор, не идти дальше.

## Фаза 2 — Shadow ML + калибровка (5–7 дней, параллельно paper)

Проверяем, что мета-лейблер реально предсказывает на СВЕЖЕЙ выборке, прежде чем
пускать его гейтить в бою.

- `ML_MODE=shadow` на отдельном контуре ИЛИ читать `/intelligence` shadow-панель;
- `ML_AUTO_RETRAIN=true`, добить выборку до ≥300 закрытых со score.

**Gate 2 (перед тем как доверять гейту в full_auto):**
- live AUC ≥ 0.63 стабильно (не разовый всплеск);
- калибровка МОНОТОННА: WR ведра 0.60–0.75 ≥ WR ведра 0.45–0.60 ≥ WR <0.45;
- ведро <0.45 стабильно худшее (это и оправдывает гейт).

Если калибровка немонотонна (как сейчас: 0.75–1.0 → WR25%) — оставляем гейт как
бинарный предохранитель, `ML_SIZE_SCALING_ENABLED=false`. НЕ включать скейлинг размера.

---

## Фаза 3 — Live limited: минимальный риск (7–10 дней)

Первые реальные ордера, но крошечным размером. Здесь важна не прибыль, а что
paper-логика совпала с реальными филлами/комиссиями/слиппеджем.

Env (по валидатору config: live требует TRADING_MODE=live/live_limited,
ROBOT_MODE!=paper, наличие TELEGRAM_BOT_TOKEN):
```
TRADING_MODE=live_limited
ENABLE_LIVE_ORDERS=true
ROBOT_MODE=live
RISK_EQUITY_USDT=<реальный минимум>   # старт с ~10–15% обычного банка
RISK_PER_TRADE_PCT=0.25               # вдвое ниже дефолта 0.5
MAX_ACTIVE_SIGNALS=2                  # с 4 → 2 на обкатку
MAX_USED_MARGIN_PCT=0.30              # с 0.85 → 0.30
```

Что сверять (paper vs live):
- расхождение цены входа/выхода (слиппедж) — в разумных пределах;
- фактические комиссии = расчётным (`cost_engine`/`_fee_rate`);
- стопы исполняются по уровню, а не «улетают»;
- Grid: включить `/grid` на 1–2 символах минимальным `GRID_BASE_ORDER_USDT`,
  проверить, что short-циклы реально ставят sell и закрываются.

**Gate 3:**
- ≥30 live-сделок;
- net PnL ≥ 0 за окно;
- медианный слиппедж вход+выход в пределах заложенного `SLIPPAGE_BUFFER_PCT`;
- ноль расхождений учёта (реализованный PnL сходится с биржей через
  `exchange_reconciliation`).

Kill-switch: дневной лосс > `ANTI_DRAIN_MAX_DAILY_LOSS_PCT` (3%) ИЛИ просадка >
`ANTI_DRAIN_MAX_DRAWDOWN_PCT` (12%) → авто-стоп уже в коде; после срабатывания —
разбор перед рестартом.

---

## Фаза 4 — Live масштабирование (постепенно)

Только после стабильного Gate 3. Наращивать по одному параметру за раз, с окном
наблюдения ≥5 дней между шагами:

1. `RISK_EQUITY_USDT` ↑ ступенями (×1.5–2 максимум за шаг);
2. `RISK_PER_TRADE_PCT` 0.25 → 0.4 → 0.5;
3. `MAX_ACTIVE_SIGNALS` 2 → 3 → 4, `MAX_USED_MARGIN_PCT` 0.30 → 0.50 → 0.85;
4. `ML_MODE=full_auto` — ТОЛЬКО если Gate 2 (монотонная калибровка, AUC≥0.63) пройден.
   Иначе остаёмся на advisory с принудительным skip.
5. `ML_SIZE_SCALING_ENABLED=true` — В САМУЮ ПОСЛЕДНЮЮ очередь, после ≥300–500
   сделок и стабильного AUC ~0.65+. До этого множитель = 1.0.

Правило одного шага: изменил один параметр — наблюдаешь окно — только потом
следующий. Любое ухудшение метрик откатывает шаг назад.

## Постоянные предохранители (действуют на всех live-фазах)

- Anti-drain: дневной лосс 3%, просадка 12%, не покупать перегретое / не шортить
  перепроданное, min RR/edge — уже в коде, не отключать.
- Re-entry cooldown: не перезаходить в ту же сторону сразу после стопа.
- Exposure/cluster guard: лимит одновременных и коррелированных позиций.
- Liquidity guard: не входить/не рубить стоп на спайке спреда.
- Grid изолирован в свой карман маржи (`GRID_MAX_USED_MARGIN_PCT=20%`), тренд не
  трогает.

## Быстрый откат (rollback)

Любая фаза, любой момент:
1. `ENABLE_LIVE_ORDERS=false` → система падает в dry_run (по валидатору config),
   реальные ордера прекращаются.
2. Бот в статус not-running / `TRADING_MODE=paper_signal`.
3. Открытые позиции — закрыть вручную через owner-эндпоинты (`/positions`, `/grid` close).
4. Все фиксы за флагами (`ML_ENFORCE_ADVISORY_SKIP`, `TREND_PEAK_LOCK_ENABLED`,
   `ML_SIZE_SCALING_ENABLED`, пороги грейдов/CVD) — откат = вернуть значение в env,
   рестарт, без релиза кода.

## Красные линии (стоп безусловно)

- Расхождение учёта PnL с биржей.
- manage-loop крашится / позиции «зависают» без ведения.
- Слиппедж систематически съедает заложенный edge (net TP1 уходит < 0).
- Просадка > 12% — не «пересиживать», разбирать причину.
