# Аудит торговой системы FINMT — 2026-07-03

Глубокий разбор: структура, бэкенд, фронтенд, торговые движки, ML.
Только факты из кода репозитория и приведённых живых тестов API.

---

## 1. Резюме (главное)

Система архитектурно зрелая (гейты, guard'ы, lifecycle, ML control-plane, fail-open), но **математически убыточная на текущих параметрах**:

| Факт | Значение | Источник |
|---|---|---|
| Winrate всех залогированных сделок | **29.31%** (51/174) | `/ml/outcomes/stats` |
| Последние 17 закрытых позиций | WR 35.3%, **сумма −4.53 USDT** | `/positions` |
| Средний win / средний loss | +2.06 / −1.54 → **expectancy ≈ −0.27 USDT/сделку** | `/positions` |
| Grid: 79 закрытых циклов | **+1.80 USDT** при запертых 189.8/190 USDT маржи | `/grid/state` |
| Grid: последние 20 циклов истории | 12 flip-закрытий = **−1.73**, 5 TP = +1.00, итог −0.77 | `/grid/state` |
| Исторический датасет (storage/ml, 101 сделка) | PnL −90.75 USDT, **суммарные издержки 93.17 USDT** | `trade_outcomes.jsonl` |
| Live shadow ML | AUC **0.7424**, гейт score≥0.45 дал бы **+10.39 USDT** на 17 сделках | ML-страница |

**Ключевой вывод:** система не «плохо предсказывает» — она **отдаёт весь edge издержкам и ранними/поздними выходами**. Исторически издержки (93 USDT) ≈ модуль убытка (91 USDT). Это лечится рефакторингом экономики, а не новыми фильтрами.

## 2. Карта системы

**Бэкенд** (`apps/api`, ~30.5k строк): FastAPI, `main.py` — монолит 3687 строк.
- `workers/robot_loop.py` — конвейер сигналов: intelligence → грейд/публикация → post-loss cooldown → depth-гейт → trade plan → performance guard → production gate → symbol policy → anti-drain → exposure → reentry cooldown → ML controller → Signal.
- `services/market_intelligence.py` (1999 строк) — MTF-анализ (1m…4h) + range/CRT/микроскальп.
- `services/signal_lifecycle.py` (1497) — ведение published→opened→tp1→closed, MFE/MAE, выходы.
- `services/exit_policy.py` — exit v4: scalp-замок, time-stop, breakeven-lock, failed_setup, трейлы.
- `services/grid_engine.py` — сетка с мартингейлом объёма, регайм-флипами и ATR-стопом.
- ML: `ml_controller` (off/shadow/advisory/full_auto) → `ml_meta_labeler` (LogReg, 15 фич) → `ml_features` → `ml_trade_logger` → `ml_shadow_report`.

**Фронтенд** (`apps/web`): Next.js, тонкий read-only дашборд через same-origin proxy (owner-токен не течёт на клиент — сделано правильно). Логики нет, риска нет.

## 3. Где теряется профит (ранжировано, по фактам)

### Утечка №1 — модель издержек ест скальпы (структурная)
Факт: AAVE #181 (скальп): notional ≈ 99.9 USDT, `total_cost = 0.45002` → **0.45% round-trip** (spot taker 0.2%×2 + слиппедж). При этом цели скальпа: `SCALP_TARGET_PCT = 0.8%`, TP1 range-профиля ≈ 0.8–1.2%. Дашборд заявляет `market: swap` (HTX swap taker = 0.05%), но paper-исполнение считает по спот-ставке 0.2%.
- `execution_engine.close_paper_position` → `cost_engine.estimate(market_type=settings.execution_market_type)` — если `ENABLE_FUTURES_EXECUTION` не включён, это `spot` независимо от баннера.
- `exit_policy._estimated_net_usdt` (строка 76) и breakeven-буфер в `signal_lifecycle` (строки 559, 1353) **жёстко используют `SPOT_TAKER_FEE`** даже для swap.
- `_net_safe_profit_pct` имеет пол 0.60% — производный от спот-комиссий; для swap он завышен в ~2 раза и сдвигает все защитные пороги.

**Издержки 0.45% при целевом ходе 0.4–0.8% = отрицательное матожидание by construction.** Это объясняет исторические −90.75 USDT при издержках 93.17.

### Утечка №2 — scalp_time_stop превращает «почти безубыток» в убыток
Факт: AAVE #181 — MFE +0.158% (не дошёл до arm 0.5%), 45 минут → закрыт по рынку **−0.576%**, `missed_profit 0.73%`, `positive_then_negative=true`. Следом post-loss cooldown 25 мин заблокировал ре-вход, а следующий шорт AAVE (#173) дал **+1.35**.
`exit_policy.before_tp1_decision`: time-stop не смотрит ни на текущий PnL, ни на поток (flow_against), ни на издержки — рубит по возрасту.

### Утечка №3 — grid: флип-чурн и мартингейл против импульса
Факты из `/grid/state`:
- 12 из последних 20 закрытий = `grid_regime_flip` с суммой **−1.73**; выигрыши TP всего +1.00. BTC за сутки сделал 4 флипа long↔short подряд, каждый с минусом.
- TRX-шорт: 5/6 уровней мартингейла залито **против** сильного импульса вверх (5m RSI 87.3, volume_ratio 5.7), позиция −0.64, цена у SL. Regime-детектор сетки — EMA200/RSI на 1h — на 10–14 часов позже реальности.
- AAVE neutral-грид: залиты buy@86.85 и sell@86.95 (flat), `unrealized −0.106` при спреде AAVE 0.1–0.5% — нейтральная сетка на неликвиде платит спред каждым кругом.
- Капиталоёмкость: 190 USDT заперто, 79 циклов → +1.80 USDT (≈0.9% за ~4 дня при полной утилизации кармана).

### Утечка №4 — ML уже видит edge, но не подключён к решению
Факты: offline-валидация слабая (val AUC 0.5411, acc 0.4528 — хуже монетки), но **live shadow AUC 0.7424**; калибровка монотонна в главном бакете: score 0.45–0.60 → WR 71.4% (+6.23), score <0.45 → WR 8.9/12.5% (−10.39). Гейт-симуляция: «взято 8 (WR 62.5%) / отрезано 9 (WR 11.1%), выгода +10.39 USDT».
Проблемы модели:
- Фичи стакана шумные: в live `cvd_trades = 1` почти везде → `cvd_ratio = ±1.0` — мусор, который модель ест как сигнал (`ml_features.py` не зануляет CVD при малой выборке, хотя `orderbook_analyzer` для гейтов это делает).
- Метка `is_win` при 29% позитивов + LogReg на 174 сэмплах → нестабильно; финальная модель обучается на всех данных без калибровки вероятностей.
- `ML_MIN_SCORE_TO_TRADE = 0.45` выбран априорно, а не из калибровочной таблицы.

### Утечка №5 — конфликт движков на одном символе
Факт: TRX одновременно — шорт intelligence-движка (позиция #174, −0.93) **и** шорт-грид 470 TRX (−0.64), при том что сам intelligence оценивает TRX как `watch_long` / trend_up. Grid изолирован «по марже», но не по риску: суммарная экспозиция на символ нигде не консолидируется.

### Утечка №6 — операционный шум и расхождения
- `intelligence_events`: **34 781 запись**; `blocked_post_loss_cooldown` и `blocked_depth_gate` пишутся каждый тик (~70 с) все 25 минут кулдауна — 6+ дублей подряд в тестовой выборке. Растёт БД, тонет полезный сигнал.
- Закрытые позиции возвращают ненулевой `unrealized_pnl` (это net_pnl закрытия, `execution_engine:293`) — вводит в заблуждение фронт/аналитику.
- `process_signal` и `process_signal_with_price` в `signal_lifecycle.py` — ~450 строк дублированного кода, **уже разошлись**: guard `low_grade_capital_release` есть только в `process_signal_with_price` (тестовый путь), в боевом `process_signal` его нет. Классический баг дивергенции.
- `_ready_candidate_check` в robot_loop дублирует всю цепочку гейтов ради подсчёта кандидатов — второй очаг дрейфа.
- Depth-гейт для range: `OB_MAX_SPREAD_PCT = 0.08%` при реальном спреде AAVE 0.1–0.52% → символ почти неторгуем скальпом, но при этом торгуется гридом без этого гейта (несогласованность политик ликвидности).
- Confidence-инфляция: `_intelligence_effective_confidence` поднимает уверенность до `max(base, setup_score*0.9)` (кэп 88) + blend с MLScorer (кэп 92). Пороги гейтов сравниваются с «накрученной» величиной — калибровка «confidence → фактический winrate» не подтверждается данными (29–35% WR при типичных confidence 60–78).

## 4. Рефакторинг и исправления (приоритезировано)

### P0 — экономика (прямой эффект на PnL)

1. **Единый CostModel.** Один сервис-источник ставок (maker/taker/slippage/funding) по `execution_market_type`, инжектится во все точки: `cost_engine`, `exit_policy` (`_estimated_net_usdt`, `_net_safe_profit_pct`), breakeven-буфер `signal_lifecycle`, grid (`GRID_FEE_ROUND_PCT`). Убрать все прямые обращения к `SPOT_TAKER_FEE`.
   Если торговля реально на swap — включить `ENABLE_FUTURES_EXECUTION` / прокинуть swap-ставки: издержки скальпа падают с 0.45% до ~0.12–0.14%, и та же книга сделок из −90.75 стала бы примерно −90.75 + (93.17 − 93.17×0.3) ≈ **+... заметно положительной по костам** (пересчитать точно скриптом по jsonl — издержки сокращаются ~70%).
2. **Maker-first вход.** Скальп/range вход — post-only лимит в зоне (сейчас paper входит по last внутри зоны как taker). На HTX swap maker 0.02% vs taker 0.05%; на споте 0.2%→0.2% (нет смысла — ещё аргумент за swap).
3. **Cost-aware scalp_time_stop.** Не закрывать по возрасту, если `current_pct > −net_safe_pct` и поток не против (`flow_against=False`); вместо мгновенного рыночного закрытия — сократить/поставить лимит у micro-VWAP. Вариант: продлевать таймер при `mfe > 0.5×arm`.
4. **Гейт для full_auto ML по калибровке.** Данные уже показывают: score<0.45 → WR ~11%. Перевести ML_MODE в `advisory` немедленно и в `full_auto` после ≥50 закрытий со score при live-AUC ≥0.6. Порог брать не константой, а из калибровочной таблицы (минимальный бакет с WR>45%).

### P1 — grid

5. **Стоимость флипа как гейт.** Флип разрешать только если `|unrealized| < ожидаемый edge нового цикла` (сейчас флип фиксирует любой минус). Добавить ATR-гистерезис: флип только при |price−EMA| > k×ATR, а не по 3 тикам подряд.
6. **Запрет мартингейла против импульса.** Не доливать уровни, если 5m/15m momentum против стороны корзины (данные уже есть в market_intelligence — переиспользовать контексты). TRX-кейс: 5 доливок в шорт при RSI(5m) 87 — прямое следствие слепоты сетки к младшим ТФ.
7. **Экономический фильтр символов сетки.** Не открывать циклы, где `шаг сетки < spread + fee_round` (AAVE neutral-грид с его спредом 0.1–0.5% — отключить). Гриду использовать тот же LiquidityGuard-порог, что и скальпу.
8. **Учёт комиссий в realized PnL сетки** (сейчас `unrealized_pnl` уровней — без fee, `GRID_FEE_ROUND_PCT` участвует только в breakeven).

### P2 — ML-слой

9. **Фичи:** занулять `cvd_ratio/cvd` при `cvd_trades < OB_CVD_MIN_TRADES` (train и serve одинаково — правка в `ml_features.row_to_features`); добавить: stop_distance_pct, ATR%(15m), volume_ratio(15m), RSI(15m), позицию цены в диапазоне, час дня (sin/cos), спред в bps.
10. **Метки:** triple-barrier (TP/SL/время) вместо бинарного `is_win`, который смешивает «−0.05% time-stop» и «−0.9% стоп» в один класс. Ваша инфраструктура lifecycle (MFE/MAE) уже пишет всё нужное.
11. **Валидация:** purged walk-forward (3–4 фолда) вместо одного хронологического сплита; изотоническая калибровка вероятностей; отчёт по-бакетно уже есть (`ml_shadow_report`) — включить его в критерий авто-промоушена shadow→advisory→full_auto (авто-эскалация по метрикам, с авто-откатом при деградации live-AUC).
12. **Разделение по стратегиям:** отдельные модели/пороги для `trade_mode=scalp` и `trend` — их распределения исходов различны by design.

### P3 — код и надёжность

13. **Слить `process_signal` / `process_signal_with_price`** в одну функцию с параметром источника цены (устранит уже существующую дивергенцию `low_grade_capital_release`).
14. **Дедупликация intelligence_events:** блок-события писать один раз на эпизод (upsert с `count`/`last_seen_at`), не каждый тик; добавить retention/архив (34.7k записей и растёт).
15. **`Position.unrealized_pnl` → 0 при закрытии**, а net PnL — в отдельное поле `realized_pnl` (API `/positions` сейчас отдаёт вводящее в заблуждение значение).
16. **Разбить `main.py` (3687 строк)** по роутерам; `_ready_candidate_check` заменить на вызов той же цепочки гейтов в dry-run режиме (один источник правды: гейт-пайплайн как список объектов `Gate.check(candidate, ctx)` — сейчас цепочка размазана по 400 строкам цикла).
17. **Кросс-движковый RiskLedger:** единый учёт экспозиции символ→(интеллект, сетка, funding-arb), лимит на суммарный нотионал и на одновременные противоположные/сонаправленные позиции разных движков.

## 5. Инновационные предложения (после P0–P1)

- **Калибратор confidence по фактам.** Вместо эвристики `max(base, setup*0.9)` — маппинг `confidence_bucket → эмпирический WR` из trade_outcomes (уже 174 строки), обновляемый еженедельно. Гейты сравнивают эмпирическую вероятность, а не инфлированную оценку. Это та же идея meta-labeling, но для rule-based ядра — ноль новых зависимостей.
- **Спред-адаптивные цели.** TP1/стоп в единицах `k×ATR(15m)` с полом `m×(spread+fee_round)` на символ — цели автоматически расширяются на неликвиде (AAVE) и сжимаются на BTC, вместо одного глобального процента.
- **Экономический A/B в paper.** Раз в сутки прогонять книгу сделок через альтернативные конфиги выходов (time-stop on/off, arm 0.3/0.5/0.7, trail 0.4/0.5/0.6) по сохранённым lifecycle-трейсам (MFE/MAE/updates уже логируются — replay почти бесплатен). Продвигать конфиг только по result-based доказательству. Каркас `symbol_policy_replay.py` уже есть — расширить на exit-параметры.
- **Funding-aware удержание для swap:** перед переносом позиции через funding-время сверять знак funding rate с позицией; отрицательный carry у скальпа — закрыть до фандинга, положительный — фактор в пользу удержания.
- **Событийный вход вместо тика раз в 60–70 с:** WS-стакан уже есть (`orderbook_feed`); триггерить проверку входа на пересечении зоны, а не по таймеру — уменьшает слиппедж входа и опоздания time-stop.
